# Script-QA — בקרת איכות לתסריטי הדרכה

סקיל המייצר **דוח מפורט לפי שקפים** על תסריט הדרכה שלם (PPTX), עם ממצאים והמלצות תיקון לפי
36 בדיקות ב-7 קבוצות. **הסקיל לא מתקן** — הוא מדווח, וההחלטה לתקן היא של מפתח/ת ההדרכה.

## עקרונות-על

1. **דיווח בלבד.** לעולם אל תערוך את קובץ ה-PPTX.
2. **מיפוי מלא — לא מדגם.** חובה לעבור על **כל** השקפים (1..N) ולדווח על **כל מופע** של כל בעיה.
   אסור לדגום שקפים, ואסור לקצר רשימות ("ועוד", "etc."). אם תסריט גדול — מחלקים לרכיבים ומכסים
   כל אחד במלואו (ראה שלב 4), לא מוותרים על כיסוי.
3. **הנח שיש בעיות.** המשימה היא לצוד אותן — לא לאשר. אם לא מצאת כלום, לא חיפשת מספיק.
4. **כל ממצא = ציטוט מהשקף + מספר-שקף + המלצה.** בלי ציטוט אין ממצא.
5. **בדיקות שיפוטיות לצד דטרמיניסטיות.** הדטרמיניסטיות רצות ב-`scan.py` (כיסוי מלא מובטח);
   השיפוטיות בקריאת-שיפוט של כל שקף.

## קובצי reference (קרא לפי הצורך)

- `references/qa-dimensions.md` ⭐ — קטלוג 36 הבדיקות: איך מזהים, חומרה, ניסוח-המלצה + טבלת כיול.
- `references/authority-sources.md` ⭐ — מקורות-הסמכות לבדיקות נכונות (#5/#6/#7) + תוויות-ביטחון.
- `references/structure-and-ids.md` — מבנה היחידה, 5–6 רכיבים (כל שיא=רכיב), סכמת ID, קיבוץ שקפי-מצב, שלמות מטא.
- `references/math-typography.md` — כללי כתיבה מתמטית (`·` לכפל, `100%`, מקפים, RTL).
- `references/language-and-register.md` ⭐ — עריכת-לשון ו-register לגיל 13–14 (#1/#21): קטלוג 6
  משפחות-ליקוי, טבלת register, וטבלת כיול. **קריאה לשונית של כל שקף מולו (שלב 5).**
- `references/banned-terminology.md` — טרמינולוגיה אסורה/מבוקרת (בדיקה מול היעד).
- `references/literacy-question.md` — הגדרת שאלה אוריינית + דרישה לכל רכיב.
- `references/journey-and-levels.md` ⭐ — מקטעי מסע-הלמידה (#28), טבלת שלב→רמה טקסונומית (#29),
  דרישות-תפקיד פר-שלב (#30), ואמיל"י — הבהרת הצורך הלימודי (#31).
- `references/report-template.md` — שלד הדוח, דרגות חומרה, render ל-HTML.

## תהליך העבודה

### 1. זיהוי קלט
המפתח/ת בדרך כלל **יעלה/תעלה את ה-PPTX לצ'אט** (לא תהיה תיקיית-תסריטים קבועה). השתמש בנתיב הקובץ
שצורף/הועלה; אם לא ברור — שאל לאיזה קובץ להתייחס. בדוק תסריט אחד בכל הרצה.

### 1.5 שער זיהוי-720 (חובה — הסקיל מכויל ל-720 בלבד)
הבדיקות מכוילות **אך ורק** לתסריטי 720/מתודיקה. אל תריץ בקרה על תסריט כללי או מפרויקט אחר בלי אישור.
זהה שהתסריט הוא 720 לפי **סימנים** — עדיף כמה שיותר:

- **שם הקובץ מכיל "יעד"** (סממן מוקדם וחזק ל-720; זמין עוד לפני חילוץ) — או שהמשתמש ציין "720"/מתודיקה במפורש.
- **אחרי החילוץ (שלב 2), אמת טביעות-אצבע בתוכן** — לפחות אחת-שתיים:
  - מזהי `methodica-{תחום}-{נושא}-NN-NN-NNN` בשקפי-החוצץ.
  - מבנה **5–6 רכיבים** עם שקפי-חוצץ (כל שאלת-שיא = רכיב עצמאי).
  - **טבלת מטא בשקף 1** ("פרטים על היעד": שם-נושא, מספר-יעד, פירוט-יעד, ID, כותב/ת, מפתח/ת, תאריך, סטטוס).
  - טרמינולוגיית 720: טקסונומיית אמ"ה / 12 פעלי-מדעים / אמיל"י / רמות (בסיסי-סטנדרטי-מתקדם).

**החלטה:**
- יש "יעד" בשם **או** נמצאה ≥1 טביעת-אצבע → זה 720, המשך רגיל.
- **אין אף סימן** → אל תריץ. אמור למשתמש: *"הקובץ לא נראה כתסריט 720 (אין 'יעד' בשם ולא זוהו מזהי
  methodica/מבנה 5–6 רכיבים/טבלת-מטא). 36 הבדיקות מכוילות ל-720 בלבד — להריץ בכל זאת?"* והמתן לאישור.

### 2. חילוץ טקסט
```bash
PYTHONUTF8=1 PYTHONIOENCODING=utf-8 python -m markitdown "<file.pptx>" -o "<out>.md"
```
- **חובה** `PYTHONUTF8=1` + דגל `-o` (לא redirect `>`) — אחרת העברית יוצאת כ-`?`.
- אם `markitdown` חסר: `pip install "markitdown[pptx]"`.
- ב-Git-Bash, `/tmp/x.md` ממופה ל-`%LOCALAPPDATA%\Temp\x.md` עבור כלי Read.
- הפלט מסומן לפי שקפים (`<!-- Slide number: N -->`), עם טבלאות, תמונות (`![alt](img)`), והערות
  דובר תחת `### Notes:`.

### 3. סריקה דטרמיניסטית מלאה (חובה)
הרץ את הסורק על הטקסט המחולץ — הוא מבטיח כיסוי **מלא** של הבדיקות המכניות (כל מופע, בכל שקף):
```bash
PYTHONUTF8=1 python "<skill>/scripts/scan.py" "<extracted.md>"
```
הסורק מחזיר, לפי שקף: רווחים כפולים + **מקף ארוך —/– (חשד-AI)** (#2), אחוז-עם-רווח/כפל-לטיני (#4), מילה כפולה (#1),
פנייה ממוגדרת (#3), חוצצי-רכיבים+מזהים (#8/#9), מפתחות-תשובה להתפלגות (#15), מקור-תבנית-פיגמה + הערות
(#11), ו**רובד לשוני (#1/#21): משפט ארוך (>25 מילים), watchlist מילים אקדמיות/מליציות, ותבניות
יתירות**. הטמע את כל מופעיו בדוח — אל תסנן ידנית. (הוא line-by-line; אל "תתקן" ממצאים שהוא לא
מצא בלי לאמת — הוא נמנע מ-false-positive כמו `%` בתא-טבלה. ה-watchlist הוא דגל-register לבדיקה, לא פסיקה.)

### 3.5 סריקה גיאומטרית + בדיקת-עין (חובה — כאן הבדיקות מפסיקות לנחש)
```bash
PYTHONUTF8=1 python "<skill>/scripts/geometry_scan.py" "<file.pptx>"
```
קורא את ה-PPTX עצמו ומחזיר, לפי שקף: **מיקום סמני-✓ ירוקים + המסיח הקרוב** (ראיה ישירה ל-#14 —
הצלב מול "התשובה הנכונה" בהערות; אי-התאמה = ממצא 🔴), אלמנטים **חוצי-גבול-קנבס** (חלקם על השקף
חלקם בחוץ — ליקוי-פריסה; אלמנט שכולו מחוץ לקנבס = חניית-תבנית תקינה, לא ממצא), ו**חשדות-גלישת-טקסט**
(טקסט גדול מהתיבה שלו).

**בדיקת-עין (vision) — חובה למסך בחירת-הדמות (#26ב):** אתר את שקף בחירת-הדמות, חלץ את התמונות
וקרא אותן בעיניים — אל תשפוט מגדר לפי שם-קובץ:
```bash
PYTHONUTF8=1 python "<skill>/scripts/geometry_scan.py" "<file.pptx>" --extract-images <N>
```
ואז Read על קובצי התמונות שחולצו. אותו כלי משמש גם לשקפים שהסריקה הגיאומטרית סימנה כחשודים.

### 4. ניתוח מבנה
קרא את `structure-and-ids.md`, ואז אתר: **טבלת מטא** (שקף 1: "פרטים על היעד" — ID, יעד, כותב/ת,
מפתח/ת, תאריך, סטטוס; **תבנית עדכנית — ללא ספירות מוצהרות**) · **5–6 שקפי-חוצץ** + IDs (כל שאלת-שיא
= רכיב עצמאי) · **קיבוץ שקפי-מצב** לפריט אחד (התחלתי/נכון/שגוי/רמז) · **הערות דובר**.

### 5. מעבר בקרה שיפוטי — כיסוי מלא של כל שקף (36 הבדיקות)

**אסור לדגום.** כל שקף 1..N נקרא ונבדק. השלם את הבדיקות שאינן מכניות (`scan.py` כיסה את המכניות):

  רמז (#17), טון (#18), אותנטיות מסיחים (#19), עומס קריאה (#25, סף >70/>120 מילים), הצג-אל-תספר (#27),
  **אותנטיות-הנושא — אמיתי ולא מומצא/מופרך (#32)**, **רלוונטיות-הנושא לגיל ז-ח (#33)**,
  **יישור השאלה ליעד שבשקף 1 (#34)**.

> **חלץ קודם את ניסוח היעד (שקף 1) — עוגן ל-#34/#35.** לפני מעבר-הבקרה קרא את **טקסט היעד** מטבלת
> המטא (שקף 1) ורשום אותו. לכל שאלה בדוק שהיא באמת נוגעת ליעד (#34); בסוף — האם היחידה כולה בונה את
> המיומנות שהיעד דורש וברמה הנדרשת (#35).

> **סוויפ לשוני ייעודי (#1/#21) — חובה, לא לדלג.** ליקויי-לשון וניסוח-לא-מותאם-גיל מדלפים דרך

> "כאילו אתה תלמיד/ת ז'" ואתר את 6 משפחות-הליקוי (יתירות · משפט מסורבל/תחביר שבור · register גבוה/נמוך ·

השקפים המלא** של הרכיב שלו (לפי החוצצים), את **ניסוח היעד** (ל-#34), את פלט `scan.py` לרכיבו,
והנחיה לקרוא את קובצי-ה-reference: `qa-dimensions.md` (חובה), `language-and-register.md` (לסוויפ
הלשוני), `math-typography.md` / `authority-sources.md` (לפי תחום), `journey-and-levels.md`,
`literacy-question.md`, `structure-and-ids.md`.

**פורמט-החזרה מחייב — JSON, לא פרוזה** (זה מה שמאפשר מיזוג/ספירה/אימות-כיסוי מכניים):
```json
{"range": [1, 30],
 "reviewed_slides": [1, 2, "...כל שקף שנקרא בפועל"],
 "findings": [
   {"slide": 17, "slides_extra": [34], "check": 29, "severity": "🔴|🟠|🔵",
    "quote": "ציטוט מילולי מהשקף", "recommendation": "המלצה מלאה, מוכנה להדבקה",
    "fixes": "מה הפתרון פותר"}],
 "passed_checks": [{"check": 5, "note": "כל התרגילים חושבו — נכונים"}]}
```
**כל ממצא = מספר-בדיקה ראשי אחד** (טבלת-ההכרעה לחפיפות: `qa-dimensions.md` › "כלל בדיקה-ראשית");
קשר לבדיקה נוספת מציינים בגוף ההמלצה, לא כממצא שני.
כל סוכן שומר את ה-JSON שלו לקובץ. לאחר סיום כולם:
```bash
PYTHONUTF8=1 python "<skill>/scripts/merge_findings.py" <N> f1.json f2.json ... -o merged.json
```
הסקריפט מחזיר: **ספירות-חומרה לכותרת (לעולם לא סופרים ידנית)**, פערי-כיסוי (שקף שאף סוכן לא
סקר → שגר סוכן-השלמה לפני כתיבת הדוח), כפילויות-גבול (הוסרו), וממצאים פסולים (בלי ציטוט/המלצה —
תקן את הסוכן, אל תמציא). הדוח נכתב **מתוך `merged.json`**, בתוספת בדיקות רמת-היחידה שמצריכות
מבט-על (#8–#26, #28–#31, #34/#35).

### 6. הפקת `report.md`
לפי `report-template.md`: כותרת + סיכום-חומרות + **שורת כיסוי** ("נסרקו N/N שקפים"), פירוט מלא
לפי-שקף (הסעיף המרכזי, כלי-העבודה), ממצאים ברמת-יחידה (חוצי-שקפים בלבד), טבלת **כיסוי מלא של 35
הבדיקות** (אינדקס).
**מיקום הפלט:** תיקיית `reports/` בתיקיית-העבודה (`reports/` בשורש הפרויקט; צור אם חסרה — **לא**
בתוך תיקיית הסקיל). שם הקובץ: `report-<שם-התסריט>.md`. (אל תניח קיום תיקיית "תסריטים לדוגמה".)
**שמור גם את `merged.json` ליד הדוח** בשם `findings-<שם-התסריט>-<YYYYMMDD>.json` — זה מה שמאפשר
בקרה-חוזרת.

**בקרה חוזרת (המשתמש מבקש לבדוק גרסה מתוקנת):** הרץ בקרה מלאה כרגיל, ואז השווה לקובץ-הממצאים
של הסבב הקודם:
```bash
PYTHONUTF8=1 python "<skill>/scripts/compare_runs.py" "<findings-ישן.json>" "<merged-חדש.json>"
```
את בלוק "דוח סבב" (תוקן / נשאר / חדש) שהסקריפט מדפיס — הדבק **בראש הדוח**, מיד אחרי שורת-הסיכום.
ההתאמה לפי ציטוט+בדיקה, כך שהזזת שקפים בין גרסאות לא שוברת את ההשוואה.
